### 3. Latency: What causes it and is yours optimized?
TypeORM (your database connector) is notoriously famous for hidden latency issues. Latency often stems from:
1. **The N+1 Query Problem:** Automatically querying the `causes` table line-by-line for every single stop, rather than using an `INNER JOIN`.
2. **Heavy Object Mapping:** TypeORM taking raw SQL rows and spending hundreds of milliseconds trying to instantiate heavy TypeScript Class Objects before returning them.
3. **Full Table Scans:** Executing a `TIME_TO_SEC` string diffing function over hundreds of thousands of un-indexed rows inside MySQL.

**Is your SQL optimized right now?**
**YES. It is extremely well optimized.** The developer specifically stripped out TypeORM's default features to squeeze out maximum speed.

Here are the specific "tricks" used in your code to eliminate latency:

1. **Bypassing the ORM entirely (`this.dataSource.query`):** Instead of using TypeORM's `.find()` or QueryBuilder which drags memory, they wrote raw MySQL directly. This drastically cuts down the JS execution time.
2. **Pre-Computed Duration (`IFNULL(s.Duree, 0)`):** Instead of making the database calculate the difference between `startTime` and `stopTime` inside the SQL statement via `TIMESTAMPDIFF`, your database actually contains a pre-calculated `Duree` column. Simply fetching a static integer rather than computing it on the fly is a massive speed boost.
3. **Raw `INNER JOIN`:** It explicitly forces MySQL to combine data on an index (`c.id = s.cause_id`) efficiently, preventing the N+1 problem.
4. **Dynamic String Construction:** It dynamically builds the `WHERE` statement array in JavaScript (`whereParts.push(...)`). This prevents SQL injection parameters while guaranteeing MySQL only evaluates conditions that are strictly necessary, allowing the database to hit its B-Tree Index quickly.
